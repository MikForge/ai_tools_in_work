# Harness Engineering 学习

## 动机 / 背景

"Harness Engineering" 指构建内部工程平台和工具链，让其他工程师更高效交付的工程实践。随着团队规模增长，靠个人纪律和文档无法保证质量，需要一套系统化的工程底座来"收束"开发流程——CI/CD、可观测性、发布系统、本地开发环境等。整理一个学习路线，建立这个方向的系统认知。

## 大致思路

| 领域 | 学习方向 |
|---|---|
| CI/CD | 流水线设计、构建缓存、增量测试、蓝绿/金丝雀发布 |
| 可观测性 | 日志/指标/链路追踪三件套、SLO/告警设计、on-call 实践 |
| 本地开发 | DevContainer、热重载、环境隔离、pre-commit hook |
| 发布工程 | 特性开关、灰度策略、回滚机制、版本管理 |
| 工程治理 | 代码规范自动化、架构约束（lint/ArchUnit）、依赖管理 |
| 内部平台 | 自服务门户、基础设施即代码、平台 API 设计 |

## 关键约束 / 注意点

- 平台团队要当成产品团队运作，运维心态做不好 Harness
- 尽量先适配现有工具，再考虑自研——非核心差异化投入不值得
- 关注 DX（开发者体验），内部工具的易用性直接影响推广成功率
