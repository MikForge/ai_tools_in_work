# Agent Pipeline 架构方案对比分析

## 背景

为 `generate-agent-workflow` 和 `agent-engineering-pipeline` 选择生成工作流 pipeline 的架构模式。评估标准基于 **Harness Engineering** 的核心理念：构建约束、反馈回路、上下文工程和熵管理，而非优化模型本身。

## Harness Engineering 完整定义

来源：Mitchell Hashimoto（2026-02-05），OpenAI 百万行代码实验（2026-02-11），Martin Fowler 分析，runoob.com 整理

### 什么是 Harness Engineering

Harness = 马具（缰绳、马鞍、马衔）。核心理念：**人类掌舵，智能体执行（Human Steer, Agent Execute）。**

Harness Engineering（驾驭工程）是围绕 AI 智能体设计和构建**约束机制、反馈回路、工作流控制和持续改进循环**的系统工程实践。它不优化模型本身，而是优化模型运行的**环境**。

Mitchell Hashimoto 的核心论断：每次 Agent 犯错，正确的响应是工程设计一个方案，让 Agent "未来不会再犯同样的错误"——把失败视为环境设计不完善的信号，而非模型能力不足。Agent 的每次失败都被视为环境设计不完善的信号，正确回应是重新设计运行环境而非换更强的模型。

### AI 工程范式的三次跃迁

| 范式 | 优化对象 | 解决的问题 | 时期 |
| ---- | -------- | --------- | ---- |
| **提示词工程（Prompt Engineering）** | 输入措辞 | 单次对话质量 | 2023~2024 |
| **上下文工程（Context Engineering）** | 信息输入 | 知识边界与幻觉 | 2025 |
| **驾驭工程（Harness Engineering）** | 运行环境 | Agent 可靠性与可持续性 | 2026~ |

类比：提示词工程是"对马喊话的技巧"；上下文工程是"给马看的地图"；驾驭工程是"给马造高速公路，配上护栏、限速牌和加油站"。

### Agent 常见失败模式

Anthropic 工程师总结了三种典型失败模式：

1. **试图一步到位（One-shotting）**：Agent 在一个会话中做完所有功能，导致上下文窗口耗尽，留下无文档的半成品代码。
2. **过早宣布胜利**：项目后期，看到部分功能完成后就宣布任务完成，忽略大量未实现功能。
3. **过早标记功能完成**：写完代码即标记完成，未做端到端测试，单元测试通过不代表功能真正可用。

此外，Agent 擅长模式复制——代码库里有什么模式就忠实复制并放大，包括坏模式和架构漂移，导致不加约束时以惊人速度积累技术债务。

### Harness 与传统框架的关系

Harness 不是 SDK、脚手架或 Agent 框架的替代品，而是**位于它们之上的一层**：

```
Harness 层（驾驭层）    → 约束、反馈循环、上下文工程、熵管理、生命周期管理
Agent 框架层            → Agent 定义、消息路由、任务生命周期（LangGraph/AutoGen/CrewAI）
SDK/API 层              → 模型调用、工具注册、流式输出（OpenAI/Anthropic/Gemini）
基础模型层              → GPT、Claude、Gemini、DeepSeek
```

传统框架解决"如何构建 AI 智能体"，驾驭层解决"智能体如何可靠地运行"。模型正逐渐吸收框架约 80% 的功能，剩余 20%——**持久化、确定性重放、成本控制、可观测性、错误恢复**——正是驾驭层的价值所在。

## 四道护栏（Guardrails）

| # | 护栏 | 含义 | 实现方式 |
|---|------|------|---------|
| G1 | **上下文工程** | 上下文是稀缺资源，按需注入不堆满窗口 | AGENTS.md 为入口，教 Agent 按需拉取额外上下文；每行对应一个历史 Agent 失败案例；文档是活的反馈循环 |
| G2 | **架构约束** | 显式分层依赖，自动强制执行 | `Types → Config → Repo → Service → Runtime → UI` 严格分层，下层不可反向依赖上层；linter 编码规则，CI 阻断违规合并；linter 报错本身也是上下文——解释 WHY 和正确做法 |
| G3 | **反馈回路** | 失败自动循环回模型，Agent 审查 Agent | Code review 从人审人变成 agent 审 agent；hooks 运行测试，失败循环回模型；AI 写的测试通过 buggy 代码时，harness 标记测试无效并迫使重新思考测试边界 |
| G4 | **熵管理** | 技术债当高利贷，持续小修取代危机清理 | 后台 Doc-gardening Agent 持续扫描文档-代码不一致，自动提交修复 PR；定期质量评分和定向重构；Agent 为 Agent 维护文档 |

**核心悖论**：更大的 AI 自主权需要更严格的运行时约束——就像高速公路护栏让你能以 120km/h 行驶。Birgitta Böckeler："增加信任需要的不是更多自由，而是更多限制。"

### 六大行业共识

1. **瓶颈在基础设施，不在模型智能**：五个独立团队得出相同结论，仅改变 Harness 工具格式即可大幅提升模型得分。
2. **文档必须是活的反馈循环**：静态文档是坟场，后台 Agent 定期清理过时文档并提交 PR。
3. **思考与执行分离**：复杂任务不能在单上下文窗口完成，需要 Orchestrator + Worker 分层架构，状态持久化至外部存储。
4. **上下文不是越多越好**：上下文是稀缺资源，巨大指令文件挤掉任务空间，应按需检索、动态注入。
5. **约束必须自动化**：人工 Review 是瓶颈，护栏编码为 Linter、CI、类型系统，由机器执行。
6. **工程师角色在转变**：从代码编写者变成环境建筑师，最大挑战是设计让 Agent 可靠工作的控制系统。

## 行为树与 Harness Engineering 的契合度——诚实的边界

以下基于 runoob.com 的完整定义重新审视：四道护栏、三种失败模式、六大行业共识、"Agent 擅长模式复制"。

### 行为树能覆盖的

| 护栏 | 行为树对应 | 覆盖程度 |
|---|---|---|
| G2 架构约束 | `Sequence` 节点强制顺序执行，下层不完成上层不动 | 覆盖了**执行顺序约束**，但未覆盖**代码依赖方向约束**（Types 不反向依赖 Runtime）——后者需要 linter/CI，不在行为树能力范围 |
| G3 反馈回路 | `Fallback` 节点表达"主方案 → 备选人工介入"，失败路径是树的一等结构 | 覆盖了**已知失败路径的事前布局**，但未覆盖**运行时自动循环回模型**——后者需要外部 hooks 和 agent review 基础设施 |
| G1 上下文工程 | `Condition` 节点满足条件才加载子阶段 | 覆盖了**按需加载的结构表达**，但未覆盖**AGENTS.md 自演化**——后者是文档工程，不是控制流 |
| G4 熵管理 | 独立子树——改一个 Worker 不动其他 | 覆盖了**局部修改隔离**，但未覆盖**后台持续扫描和自动修复**——后者是持续进程，不是有限工作流 |

### 行为树如何应对三种失败模式

| 失败模式 | 行为树能否防住 | 分析 |
|---|---|---|
| **一步到位（One-shotting）** | 能 | 树的层级分解本身就是反 one-shotting 的——`Sequence` 强制拆成多阶段，每个子树独立执行，不可能在一个会话里塞完所有功能 |
| **过早宣布胜利** | 部分 | 行为树可以在每个阶段末尾设 `Condition` 门禁（如"所有测试通过"），但这取决于树的设计者是否有意识加入——树结构本身不强制写验证门 |
| **过早标记功能完成** | 部分 | 同上，需要显式的端到端验证节点。`Gate Checklist` 模式可以补这个缺口——在树的叶节点加入 checklist 验证 |

### 一个必须正视的新风险：Agent 擅长模式复制

runoob.com 明确指出：**Agent 擅长模式复制——代码库里有什么模式就忠实复制并放大，包括坏模式和架构漂移。**

这对行为树意味着什么？如果 `generate-agent-workflow` 这个 agent 本身用行为树生成工作流，它会在项目里复制**现有的工作流模式**。如果已有 workflow 是松散的线性脚本，agent 生成的"行为树"也只会是线性序列套一层树壳——挂了行为树的名，没有行为树的实。

**这不是行为树作为模式的缺陷，而是 agent 驱动生成的通病。** 缓解措施：

- 行为树的 schema/模板必须是硬约束，不允许生成变体偏离核心语义（`Sequence` 必须顺序、`Fallback` 必须有备选）
- 需要类似 linter 的验证：生成的树是否符合行为树的结构规范，还是只是线性序列换皮

### 六大共识下的重新审视

| # | 共识 | 对行为树的影响 |
| --- | ---- | ----------- |
| 1 | 瓶颈在基础设施，不在模型智能 | 行为树就是基础设施——它为 agent 提供结构化的执行环境，降低对模型推理能力的依赖 |
| 2 | 文档必须是活的反馈循环 | 行为树不碰这个，但生成的 workflow 产物（`.md` 文件）可以成为活文档的一部分 |
| 3 | 思考与执行分离（Orcestrator+Worker） | 行为树的结构就是这种分离：**树 = Orchestrator，叶 = Worker**。这是最强的对齐点 |
| 4 | 上下文按需检索 | `Condition` 节点天然支持——不满足条件的子树整个跳过，不加载其上下文 |
| 5 | 约束必须自动化 | 行为树本身不自动化约束，但行为树的 schema 可以编码为约束（类似 linter 规则） |
| 6 | 工程师角色转变 | 行为树让工程师从"写执行步骤"转为"设计执行结构"——符合从执行者到架构师的转变 |

### 行为树无法覆盖的

| 系统级能力 | 为什么行为树不碰 |
|---|---|
| **自动化 linter 约束** | 树是工作流描述，不是静态分析规则。`Sequence` 建立执行护栏，但无法阻断不符合依赖方向的代码合并 |
| **后台持续熵管理 Agent** | 树是有起点有终点的任务结构，不适合"永远运行，持续扫描"的守护进程 |
| **AGENTS.md 根据失败案例演化** | 行为树是 agent 执行的产物，不是 agent 进入项目时读取的文档系统 |
| **Agent review agent 的审查闭环** | 审查是另一棵独立的树，需要外部触发（GitHub webhook / CI），不是在当前树内部处理的 |

### 结论

行为树与 Harness Engineering 的契合度**没有因为完整定义而削弱，反而在几个关键点上加强了**：

- **六大共识第 3 条**（思考与执行分离）直接对应行为树的 Orch/Worker 结构——这是最根本的对齐
- **三种失败模式中的 one-shotting**，行为树通过强制分解天然防止——这不是巧合，是结构设计的结果
- **"模型吸收框架 80% 功能"**意味着行为树作为**结构描述**（而非框架）的角色更重要：模型处理 HOW，树定义 WHAT
- **Agent 擅长模式复制**是一个新增的风险，但可以通过行为树 schema 硬约束 + 生成后 lint 来缓解

关系不变：**行为树 ⊂ Harness Engineering**，行为树覆盖"单次任务结构化执行"，不覆盖"系统级持续约束"。但完整定义让这个边界更清晰了——不是模糊的分工，而是明确的两层。

### 为什么是行为树——从 Harness 四道护栏出发

1. **G2 架构约束**：`Sequence` = 护栏的执行链。`Sequence` 的子节点必须从左到右逐个通过，一个失败整棵树停住——这比文档里的"建议顺序"要硬。

2. **G3 反馈回路**：`Fallback` = 反馈回路的结构化表达。主方案 → 备选方案 → 人工介入，三层降级，在树里是一等结构，不需要额外标注。

3. **G1 上下文工程**：`Condition` = 按需注入的入口。满足条件才进入子节点，不满足直接跳过返回 Success——上下文只在需要时加载。

4. **G4 熵管理**：独立子树 = 熵管理。改一个 Worker 子树，不影响兄弟节点。树的层级边界天然隔离变更。

5. **G2+G3**：树 = Orchestrator，叶 = Worker。原文明确要求"Orchestrator + Worker 分层架构"和"思考与执行分离"——行为树的结构就是这种分离。

行为树在 workflow 结构化执行层面是最优解。系统级能力（linter 自动化、后台 doc-gardening、AGENTS.md 演化）不在它的范围内，也不应该让它去覆盖。
