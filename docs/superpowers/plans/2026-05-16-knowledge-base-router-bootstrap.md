# Knowledge Base Router

## Overview

`knowledge-base-router` is the single public entry point for the project knowledge base system. Every external knowledge-base request — from a user, agent, workflow, or other skill — MUST enter through this router. The router performs environment assessment (Bootstrap Gate), intent classification, and routing orchestration. It does NOT generate content, read full documents, write files, or perform governance repairs.

Core responsibilities:

1. Execute Bootstrap Gate before any action.
2. Route user or agent knowledge-base intent to the correct internal worker.
3. Block normal read/write when state is Empty, Partial, or Broken.
4. Ask at most one clarifying question at a time, only when ambiguity blocks routing.

## When to Use

**Trigger:** A user, agent, workflow, or other skill makes a knowledge-base related request — read, write, publish, audit, maintain, or initialize — or needs to check knowledge-base state before acting.

**Do NOT re-enter the router when:** Already inside a router orchestration chain with a valid `Worker Handoff Payload` generated by this router.

### Known Rationalizations (Rejected Shortcuts)

| Rationalization | Why It's Wrong |
|---|---|
| "Just a quick lookup, no need to check state" | Even reads require Bootstrap Gate — a Partial index means the document may not be discoverable. |
| "I can just scan the directory to see what's there" | `knowledge-base-context` discovers through indexes, never by directory scan. Bypassing the index chain breaks the contract. |
| "I'll write the file now and fix the index later" | Orphan documents are exactly what this creates. Publisher is the only writer; auditor+gardener fix orphans after the fact. |

### Example Scenario

No `.knowledge-base.yml` and no default root. User says "查一下项目知识库里有没有构建配置" (check if the project knowledge base has build config docs).

Router: Bootstrap Gate → Empty. Asks "知识库尚未初始化，是否现在初始化？" (Knowledge base not initialized. Initialize now?) — does NOT scan directories, guess paths, or attempt to read.

## Bootstrap Gate

Before ANY action, the router collects lightweight evidence to determine the knowledge-base environment state. The gate produces one of four states:

| State | Condition | Route |
|---|---|---|
| Ready | `.knowledge-base.yml` exists and parses; `root`, `index`, `categories` resolvable; key indexes reachable | Normal routing |
| Empty | `.knowledge-base.yml` does not exist AND default root does not exist | If user explicitly requests init → `knowledge-base-init`; otherwise ask whether to initialize |
| Partial | Config exists but directories are missing, OR directories exist but config is missing, OR root/index/category/layer indexes are missing | `knowledge-base-auditor` (report-only); auditor determines init-compatible or repair-required |
| Broken | YAML parse failure, path escapes, category path conflicts, invalid config fields, or index paths pointing outside root | `knowledge-base-auditor` (report-only) |

### Ready Minimum Criteria

All of the following must be true:

1. `.knowledge-base.yml` parses successfully.
2. `knowledge_base.root` directory exists.
3. Root index file exists at `{root}/{index}`.
4. `categories` is non-empty.
5. Every category has `name`, `path`, `index`, `description`.
6. Every category directory and category index exists.
7. Every category path's corresponding layer index exists at `{root}/{layer}/README.md`.

### Partial/Broken Reporting Rules

- Missing config, missing root, missing root index, or missing category indexes → `Severity=blocking`.
- Broken links, orphan documents, missing summaries → `Severity=warning` or `info` (auditor determines).
- Router outputs only the state (`Ready`, `Empty`, `Partial`, `Broken`). It does NOT determine Partial subtypes or generate audit report bodies.
- Missing index files → `Partial`. Unparseable config, path escapes, or internal config conflicts → `Broken`.
- Only `knowledge-base-auditor` may classify a Partial state as init-compatible or repair-required, based on content candidates, unknown files, orphan documents, and overwrite risk.

### Partial Subtypes (Determined by Auditor, NOT Router)

| Subtype | Determination | Next Step |
|---|---|---|
| init-compatible Partial | Only init scaffold is missing, all target write paths are absent, or default root is empty/scaffold-only | After auditor report, `knowledge-base-init` may fill in |
| repair-required Partial | Existing content, unknown files, conflicting indexes, config fixes, migrations, orphan documents, or any case where existing files would be overwritten | After auditor report, only `knowledge-base-gardener` may repair within confirmed scope |

## Bootstrap Gate Evidence

Router collects ONLY lightweight evidence. It MUST NOT generate full audit reports, scan content directories, or determine Partial subtypes.

### Allowed Evidence Collection

- Whether `.knowledge-base.yml` exists.
- Whether YAML parses; if not, record the parse error summary.
- Whether `knowledge_base.root`, `knowledge_base.index`, `categories` exist.
- Path reachability of: root, root index, layer indexes, category directories, category indexes.
- List of missing or conflicting paths.

### Forbidden Evidence Collection

- Do NOT scan category directories for content documents.
- Do NOT read document full text.
- Do NOT judge duplication, staleness, factual drift, or content quality.
- Do NOT generate Audit Report Protocol bodies. These belong to `knowledge-base-auditor`.
```

## External Entry Rules

1. User natural-language requests MUST enter through the router.
2. When an agent spontaneously needs to read, write, persist, or maintain knowledge-base content, it MUST enter through the router.
3. Other workflows or skills that need to use the project knowledge base MUST enter through the router.
4. Internal workers MUST NOT start in response to an external user or agent request without going through the router.
5. Internal workers MAY output a `Handoff Recommendation`, but MUST NOT self-start the next worker.
6. The next worker in a chain MAY proceed ONLY when the router has generated a complete `Worker Handoff Payload` and required user confirmations are satisfied. Otherwise, return to the router or request user clarification.

## Internal Worker Loading Protocol

Internal workers are skill-shaped internal instruction modules within the `knowledge-base-router` package. Router starts workers using an explicit loading protocol — it does NOT rely on the model or runtime to auto-select workers from natural language.

### Loading Rules

1. Router resolves the target worker's `SKILL.md` at the package-relative path: `<target_worker>/SKILL.md` (e.g., `knowledge-base-context/SKILL.md`).
2. If Chinese behavior guidance is needed, router also reads the worker's package-relative `zh-CN.md`.
3. The physical path in this repository is `.agents/skills/knowledge-base-router/<target_worker>/SKILL.md`. Specs and worker instructions MUST NOT depend on absolute paths, `.codex/skills` symlinks, or `.claude/skills` symlinks.
4. All workers MUST read and comply with `references/contract.md`. This is a shared reference within the package — it is NOT copied as an independent contract skill.
5. Router MUST pass a complete `Worker Handoff Payload` block to the worker.
6. The worker MUST validate the payload before execution. If validation fails, return only the missing fields or rejection reason. Do NOT execute content reads, writes, or repairs.
7. Router MUST NOT start an internal worker via user original text, skill name guessing, or implicit invocation.

### Invocation Control Fields

- `agents/openai.yaml` and Claude Code frontmatter fields are guardrails to prevent external implicit triggering. They are NOT the sole mechanism for router-to-worker invocation.
- Even if the runtime does not support explicit sub-skill invocation, the router MUST proceed with orchestration after loading the worker instructions and shared reference by path.
- If an internal worker is triggered directly (bypassing the router), it MUST NOT bootstrap itself as an entry point. It MUST instruct the caller to enter through `knowledge-base-router`.

## Clarification Rules

- User says "write to knowledge base" but provides no content → route to author to generate a draft first.
- User provides a ready draft and asks to save → route to publisher.
- User says "clean up the knowledge base" → route to auditor first, never directly to gardener.
- User only says "knowledge base" with no operation type → ask exactly ONE question to clarify.
- Classification ambiguity in Ready state → ask exactly ONE question only if the ambiguity blocks routing or context scope. Publishing classification ambiguity is handled by publisher based on `category_hint`, config, and user confirmation.
```

Append to `.agents/skills/knowledge-base-router/SKILL.md`:

```markdown
## Worker Handoff Payload Contract

Every `Worker Handoff Payload` MUST contain both common fields and target-worker-specific fields. When fields are missing, the router MUST NOT guess. It may only: ask one question, fill in based on upstream recommendation, or route to auditor.

### Common Fields

| Field | Required | Description |
| ----- | -------- | ----------- |
| `source` | yes | Direct sender of this executable payload. MUST be `knowledge-base-router`. |
| `request_origin` | yes | Original external initiator: `user`, `agent`, `workflow`, or `skill`. MUST NOT be overwritten once inside the worker chain. |
| `recommended_by` | no | Upstream recommendation source that triggered this handoff (e.g., `knowledge-base-author`, `knowledge-base-auditor`). Does NOT replace `source`. |
| `intent` | yes | `init`, `read`, `author`, `publish`, `audit`, `maintain` |
| `kb_state` | yes | Bootstrap Gate output: `Ready`, `Empty`, `Partial`, `Broken` |
| `target_worker` | yes | Name of the next internal worker |
| `original_request` | yes | Summary of the original request from user, agent, workflow, or skill |
| `evidence` | no | Bootstrap Gate lightweight evidence. MUST NOT contain full document text. |
| `confirmation_status` | yes | `not_required`, `required`, `confirmed` |
| `missing_fields` | no | List of missing fields. If non-empty, worker MUST NOT execute. |

### Worker-Specific Fields

| target_worker | Required Fields | Optional Fields |
|---|---|---|
| `knowledge-base-init` | `init_mode` | `default_root`, `config_template` |
| `knowledge-base-context` | `mode` | `query`, `category`, `path`, `max_full_documents` |
| `knowledge-base-author` | `source_material`, `writing_goal` | `context_documents`, `revision_target`, `suggested_category` |
| `knowledge-base-publisher` | `operation`, `draft_or_target` | `category_hint`, `filename`, `summary` |
| `knowledge-base-auditor` | `audit_reason` | `focus_paths`, `gate_evidence` |
| `knowledge-base-gardener` | `audit_report`, `approved_scope` | `report_items`, `mode` |

### Field Rules

1. `init_mode` MUST be `empty` or `init-compatible-partial`. Ready, Broken, or repair-required Partial MUST NOT handoff to init.
2. `source` MUST be `knowledge-base-router`. Workers MUST NOT emit executable `Worker Handoff Payload`.
3. `recommended_by` records the upstream recommendation source. When a worker recommends the next step, it may only appear in `Handoff Recommendation` or let the router write `recommended_by`. It MUST NOT overwrite `source`.
4. `request_origin` records the initial initiator. Workers MUST NOT overwrite it when recommending next steps.
5. Workers needing Bootstrap Gate state read the common `kb_state` field. Do NOT create a separate `bootstrap_state` field.
6. `confirmation_status=required` means confirmation MUST be obtained before execution. Only `not_required` or `confirmed` may proceed to handoff.
7. If `missing_fields` is non-empty, worker MUST NOT execute.
8. `knowledge-base-context.mode` MUST be one of: `browse`, `search`, `category-search`, `exact-path`.
9. `mode=search` requires `query`. `mode=category-search` requires `query` AND `category`. `mode=exact-path` requires `path`.
10. `max_full_documents` defaults to 3. Router MUST NOT inflate this to unlimited.
11. `writing_goal=revise` requires `revision_target`.
12. `knowledge-base-publisher.operation` MUST be one of: `add`, `update`, `rename`, `index-only`.
13. `operation=add` requires draft content AND confirmed publish intent. Router MAY pass `category_hint`, but the final category is determined by publisher based on config and user confirmation.
14. `operation=update`, `rename`, or `index-only` requires a target file or index path.
15. `operation=index-only` is ONLY for index backfill within the current publish/update/rename chain. Existing unindexed Markdown is an Orphan and MUST go through auditor → gardener.
16. `knowledge-base-publisher.confirmation_status` MUST be `confirmed` before any write (add, overwrite, update, rename, or index-only).
17. `knowledge-base-gardener.mode` defaults to `dry-run` when not explicitly set to `apply`.
18. `knowledge-base-gardener.approved_scope` MUST NOT exceed the `Suggested Gardener Scope` from the Audit Report Protocol.
19. `knowledge-base-author`'s output category is a suggestion only. It MUST NOT be used as publisher's final classification confirmation.

### Payload Block Format

Router MUST pass the payload to the worker using the following block format. All common required fields MUST be present. `missing_fields` SHOULD always appear (empty as `[]`). `evidence` appears when there is evidence. In `worker_payload`, only include the target worker's specific fields — required fields MUST be present, optional fields only when they have values or need explicit empty expression. Never fill in fake `null` to pad fields.

````markdown
## Worker Handoff Payload

```yaml
source: knowledge-base-router
request_origin: user
intent: read
kb_state: Ready
target_worker: knowledge-base-context
original_request: "User wants to check if the project knowledge base has build config conventions"
evidence:
  - ".knowledge-base.yml parsed"
  - "root index reachable: docs/00-project-knowledge-base/README.md"
confirmation_status: not_required
missing_fields: []
worker_payload:
  mode: search
  query: "build config"
  max_full_documents: 3
```
````

`worker_payload` contains ONLY the target worker's specific fields. Common fields MUST NOT be duplicated inside `worker_payload`.

### Handoff Recommendation Format

Workers output recommendations using this format:

```markdown
## Handoff Recommendation

- From: <worker>
- Recommended next worker: <worker>
- Reason: <why this next step is needed>
- Payload delta:
  - <field>: <value or missing>
- Requires user confirmation: yes | no
- Missing fields:
  - <field>
```

A `Handoff Recommendation` is NOT an execution authorization. The router MUST check the payload, Bootstrap Gate state, and confirmation status before starting the next worker.

When a worker outputs a `Handoff Recommendation`, it does NOT need to rewrite `request_origin`. The router MUST inherit the original `request_origin` from the current payload, keep `source=knowledge-base-router`, and write `recommended_by` and new `worker_payload` as needed.
```

```markdown
## Common Mistakes

1. **Routing a non-Empty init request directly to `knowledge-base-init`.** Partial and Broken init requests MUST go through auditor first to determine init-compatible vs repair-required.
2. **Scanning category directories or reading full document text to determine Partial subtypes.** Router collects only lightweight evidence. Subtype determination belongs to auditor.
3. **Mixing `bootstrap_state`, `state`, `kb_status`, or other ad-hoc field names into the payload**, creating inconsistency with `kb_state`. Use ONLY `kb_state`.
4. **Starting publisher directly after author outputs a draft**, without confirming `operation`, `draft_or_target`, and `confirmation_status`.
5. **Treating `category_hint` as the final classification.** Final classification MUST be determined by publisher based on config and user confirmation.
6. **Starting gardener directly when the user says "clean up the knowledge base"**, without an auditor report and user-confirmed scope.
7. **Treating `missing_fields` as hints that can be self-filled.** If `missing_fields` is non-empty, worker MUST NOT execute.
8. **Relying on model implicit invocation to select an internal worker**, instead of the router loading the target worker by path.
9. **Internal worker self-bootstrapping the Bootstrap Gate or continuing execution when triggered directly from outside.**
10. **Overwriting `request_origin` inside the worker chain**, causing the original user, agent, or workflow source to be lost.

## Error Handling

| Scenario | Behavior |
|---|---|
| Empty + read/write request | Ask whether to initialize. Do NOT auto-create. |
| Empty + init request | Route to `knowledge-base-init`. |
| Partial + init request | Route to `knowledge-base-auditor` first to determine init-compatible. |
| Broken + init request | Route to `knowledge-base-auditor`. Do NOT enter init. |
| Partial/Broken + any read/write request | Block read/write. Route to `knowledge-base-auditor`. |
| Ready + classification ambiguity | Ask exactly ONE question only if ambiguity blocks routing or context scope. Publishing classification ambiguity is handled by publisher. |
| Ready + target file conflict | Route to publisher to handle conflict confirmation. |
| `Worker Handoff Payload` missing required fields | Do NOT execute worker. List missing fields. If needed, ask user one question, or fill from upstream recommendation and regenerate payload. |
| Worker recommends next step but user confirmation is missing | Do NOT proceed. Request confirmation first. |

## Acceptance Scenarios

1. No config, no directory. User says "查项目知识库" → Router asks whether to initialize.
2. No config, no directory. User says "初始化知识库" → Router routes to init.
3. Config exists but root index is missing. User says "保存文档" → Router routes to auditor, does NOT write files.
4. Ready state. User says "有没有构建配置" → Router routes to context.
5. Ready state. User says "整理一下知识库" → Router routes to auditor, NOT directly to gardener.
6. Default root exists but no config → Router does NOT directly init. Routes to auditor to determine init-compatible Partial.
7. Agent needs project KB context → Router performs Bootstrap Gate, then routes to context with read/search payload.
8. User asks to save a document → Router performs Bootstrap Gate, checks publisher payload for confirmed draft, operation, and confirmation status.

11. Partial state → Router does NOT scan content directories. Passes only lightweight evidence to auditor for subtype determination.

13. Internal worker triggered directly by user or agent → Worker does NOT execute the task. Explains that the request must enter through `knowledge-base-router` and lists missing handoff fields.
14. Agent wants to persist current conclusions → Router routes to author first to generate draft. Does NOT proceed to publisher without user confirmation.
