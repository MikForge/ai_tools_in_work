# Harness Engineering（驾驭工程）

> 来源：[菜鸟教程 - Harness Engineering](https://www.runoob.com/ai-agent/harness-engineering.html)

## 什么是 Harness Engineering

Harness Engineering（驾驭工程）由 HashiCorp 联合创始人 Mitchell Hashimoto 于 2026 年 2 月 5 日首次提出。它是一个围绕 AI 智能体构建**约束机制、反馈回路、工作流控制和持续改进循环**的系统工程实践。其核心理念是：**人类掌舵，智能体执行（Human Steer, Agent Execute）**。

六天后（2 月 11 日），OpenAI 在公布其百万行代码实验结果时正式采用了这个术语，随后 Martin Fowler 发表了深度分析。

Mitchell Hashimoto 的原话概括了这一思想：

> "Harness engineering is the idea that anytime you find an agent makes a mistake, you take the time to engineer a solution such that the agent will not make that mistake again in the future."

背后的核心信息是：**智能体的每一次失败，都暴露了环境设计的缺陷。** 应对方式不是更换更强大的模型，而是重新设计智能体的运行环境。

## 为什么需要 Harness Engineering——真实数据

- OpenAI 的团队在 5 个月内生成了约 100 万行代码，而工程师手写了 **0 行**。
- 一个 3 到 7 人的小团队，人均每天提交 **3.5 个 PR**。
- LangChain 的 Terminal Bench 2.0 从第 30 名跃升至第 5 名（得分从 52.8% 提高到 66.5%），**仅通过 Harness 改进，未更改任何模型参数**。
- 五个独立团队得出了相同的结论：**瓶颈不在模型智能，而在基础设施。**

## AI 工程的三次范式转移

| 阶段 | 时间 | 核心 | 解决的问题 |
| ---- | ---- | ---- | ---------- |
| Prompt Engineering | 2023–2024 | 优化输入措辞 | 单次对话质量 |
| Context Engineering | 2025 | 优化信息注入 | 知识边界与幻觉问题 |
| Harness Engineering | 2026– | 优化运行环境 | Agent 可靠性与可持续性 |

一个形象比喻：Prompt Engineering 是"对马喊话的技巧"；Context Engineering 是"给马看的地图"；而 Harness Engineering 是"给马造一条高速公路，配上护栏、限速牌和加油站"。

## Agent 的常见失败模式

根据 Anthropic 长期运行智能体的经验，主要有三种典型失败模式：

1. **一步到位（One-shotting）**：智能体试图在一个会话中完成所有功能，耗尽上下文窗口，最终留下一堆无文档的半成品代码。
2. **过早宣布胜利（Premature victory declaration）**：在项目后期，智能体看到部分功能有进展，就宣布任务已完成。
3. **过早标记功能完成（Premature feature sign-off）**：智能体在写完代码后立即标记功能完成，却从未进行端到端测试。

还有一个很危险的特点：智能体极度擅长**模式复制**——包括复制坏模式和错误的架构模式。这意味着，没有约束的智能体会以可怕的速度积累技术债务。

## 驾驭工程的四道护栏

### 护栏一：上下文工程（给新员工的入职手册）

AGENTS.md 是智能体进入仓库后遇到的第一份文档。但要注意，上下文是一种稀缺资源——"过多的指导反而会挤掉任务、代码和相关文档的空间"。

更好的做法是：提供一个稳定、极简的入口点，教会智能体在需要时**按需检索**额外信息，而不是一次性把所有上下文塞满。就像新员工入职时不需要读完 1000 页的流程文档一样，你只需要在正确的时刻提供正确的信息。

在 Mitchell Hashimoto 的 Ghostty 项目中，AGENTS.md 的每一行都对应了一个历史上的 Agent 失败案例——**文档是活的反馈循环，不是静态制品**。

### 护栏二：架构约束（缰绳）

OpenAI 的团队构建了一个严格的分层依赖模型：

```text
Types → Config → Repo → Service → Runtime → UI
```

**不允许反向依赖**，这些规则被编码成自定义的 Linter，一旦违反，CI 就会阻断合并。一个关键设计细节：Linter 输出的报错信息本身也是一种上下文工程——它不仅告诉智能体"你违反了什么规则"，还解释了**为什么**这条规则存在，以及**如何修复**。这让智能体能够在没有人类干预的情况下自纠正。

### 护栏三：反馈回路（Agent 审 Agent）

代码审查从"人对人"变成"Agent 对 Agent"。Codex 在本地审查自己的改动，并循环请求更多审查，直到通过。Hooks 运行预定义的测试套件，任何失败都会把错误信号循环回模型。

更关键的是：如果 AI 写的测试居然通过了有 bug 的代码，那么 Harness 就会判定这个测试无效，强制 Agent 重新考虑它的测试边界。

### 护栏四：熵管理（垃圾回收）

受 OpenAI 做法的启发：持续进行小规模的"债务偿还"，而不是积累到一定程度再批量清理——他们把技术债务称为"高利贷"。具体措施包括：

- 在后台运行 Codex 任务，不断扫描代码库中的偏离项，更新每个模块的质量评分。
- 设立专门的**文档园丁 Agent（Doc-gardening Agent）**，自动扫描代码中的变更是否与文档一致，并提交修复 PR。Agent 维护文档，给其他 Agent 使用。

驾驭工程的核心悖论，用 Birgitta Böckeler 的话说就是：

> "为了获得更高的 AI 自主性，运行时必须受到更严格的约束。"

## 六大行业共识

| # | 共识 | 核心含义 |
| -- | ---- | -------- |
| 1 | 瓶颈在基础设施，不在模型智能 | 五个独立团队得出一致结论 |
| 2 | 文档必须是活的反馈循环 | 静态文档是"坟场"；通过后台 Agent 定期清理、更新并提交修复 PR |
| 3 | 思考与执行分离 | 复杂任务需要 Orchestrator + Worker 的架构，并依赖外部状态持久化 |
| 4 | 上下文不是越多越好 | 上下文是稀缺资源，应该动态按需检索，而不是一股脑全塞进去 |
| 5 | 约束必须自动化 | 人类审查是真正的瓶颈；必须把约束编码为 Linter、CI 阻断等自动机制 |
| 6 | 工程师的角色正在转变 | 从代码的"编写者"转变为运行环境的"建筑师" |

## Harness 与传统框架的关系

Harness Engineering 不是要取代现有的 SDK、脚手架或 Agent 框架，而是位于它们**之上**的额外一层：

```
Harness 层       → 约束、反馈循环、上下文工程、熵管理、生命周期管理
Agent 框架层     → LangGraph、AutoGen、CrewAI——Agent 定义、消息路由、任务生命周期
SDK/API 层       → 模型调用、工具注册、流式输出
基础模型层       → GPT、Claude、Gemini、DeepSeek
```

传统 Agent 框架解决的是**"如何构建 AI 智能体"**的问题，而 Harness 层解决的是一个完全不同的根本问题：**"智能体如何可靠地运行"**。

模型正在吸收大约 80% 的现有框架功能，但剩下的那 20% 才真正重要：**持久化、确定性重放、成本控制、可观测性、错误恢复**——这些也是 Harness 框架能够提供持久价值的领域。

## 核心组件速查

| 组件 | 解决的问题 | 代表实践 |
| ---- | ---------- | -------- |
| 上下文工程 | Agent 不知道看什么、怎么找 | AGENTS.md 活文档，按需检索 |
| 架构约束 | Agent 复制并放大坏模式 | 分层依赖、自定义 Linter、CI 强制 |
| 反馈回路 | Agent 不知道自己错了 | Agent-Agent review、自动测试套件 |
| 熵管理 | 技术债和文档腐烂 | Doc-gardening Agent、持续垃圾回收 |

---

未来的软件开发的真正瓶颈，可能已经不再是"我们写代码能写多快"，而是**"我们能设计多聪明、多鲁棒的系统，来驾驭 AI 代理的巨大能量"**。工程师的角色，从**执行者**转变成了**赋能者和系统思考者**。
