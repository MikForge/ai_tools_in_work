# Directory to Technical Blog

## Overview

将项目目录/源码文件夹分析后，生成一篇结构完整、可直接发布到技术博客平台（掘金/CSDN/Medium/个人博客）的 Markdown 文章。

**核心原则：** 博客不是目录树 dump，也不是 API 文档。它是一篇有叙事、有观点、有代码佐证的 technical story。

**篇幅约束：** 150-300 行 Markdown（约 2000-4000 字），最多 6 个 `##` 章节。超过这个范围要么是选题太大、要么是面面俱到。宁可写两篇聚焦的文章，不写一篇流水账。

## When to Use

- 用户说"根据这个项目写篇博客"、"把这个模块整理成文章"、"生成技术博文"
- 用户提供了一个目录路径，希望产出可发布的 Markdown
- 用户想向他人介绍某个项目的架构设计

**不适用：**
- 纯 API 文档/参考手册（那不是博客）
- README 或项目文档（格式不同）
- 代码注释生成

## 必要输入

启动此 skill 前，用户必须提供一个明确的**分析目标目录**：

- **绝对路径** `/path/to/project` 或**相对路径** `./src/module`
- 拒绝模糊说法：可以说"这个项目"，但不能说"随便写一篇"
- 用户未提供目录时，主动询问："你想分析哪个目录？"

**为什么必须？** 没有目标目录，Step 1（分析目录）无法执行，后续所有步骤都是空中楼阁。这不是可选参数——缺少它 skill 不能启动。

## 五步流程

复制此清单跟踪进度：

```
写作进度：
- [ ] Step 1: 分析目录 — 理解项目做什么、怎么做的
- [ ] Step 2: 确定选题 — 找到值得讲的"故事"
- [ ] Step 3: 选择结构 — 匹配文章模板
- [ ] Step 4: 撰写正文 — 逐节写作
- [ ] Step 5: 审校输出 — 检查质量
```

### Step 1: 分析目录

**先理解，再动笔。** 至少读 3-5 个关键文件后再确定文章方向。

1. 列出目录结构（`ls -R` 或 `find`，控制深度 2-3 层）
2. 找到入口文件、核心类、关键接口
3. 识别架构模式（MVC / ECS / 插件系统 / 管道 / 状态机 ...）
4. 找到模块边界和模块间通信方式
5. 留意命名约定、设计模式、非显而易见的实现选择

**这个阶段只理解，不输出。**

### Step 2: 确定选题

从分析结果中提取 **一个** 值得讲的故事。好的选题满足：

- 有明确的"问题 → 方案"主线
- 包含可展示的代码片段
- 读者看完能学到可复用的思路

**常见选题角度：**

| 角度 | 适用场景 | 示例标题 |
|------|---------|---------|
| 架构设计 | 项目有清晰的分层/模块化设计 | "用 PureMVC 重构 Cocos Creator 项目：分层架构实践" |
| 踩坑记录 | 实现中遇到了非显而易见的坑 | "Cocos Creator UI 系统的 5 个坑及解决方案" |
| 源码解读 | 分析一个模块的内部实现 | "深入 UIMgr：Cocos Creator 中的 UI 栈管理" |
| 最佳实践 | 总结了可复用的模式 | "TypeScript 游戏开发的 10 个设计模式" |
| 工具链 | 构建/调试/部署工具 | "用 module-indexer 自动生成 barrel index" |

**选题检查：** 能用一句话说出这篇文章的"一句话价值"吗？如果不能，回到 Step 1 再读代码。

### Step 3: 选择结构

根据选题角度匹配文章模板：

**模板 A — 架构设计类：**

```markdown
# [标题]

## 背景
- 项目/模块解决什么问题
- 为什么不满足于现有方案

## 整体架构
- 架构图（用 mermaid 或 ASCII art）
- 分层/分模块说明（每层一句话职责）

## 核心设计
- 选 2-3 个最有价值的设计点
- 每点：问题 → 方案 → 代码示例 → 效果

## 踩坑与取舍
- 做过的 trade-off
- 为什么选了 A 不选 B

## 总结
- 适用场景
- 一句话核心价值
```

**模板 B — 实现细节/源码解读类：**

```markdown
# [标题]

## 引言
- 这个模块做什么，为什么值得深入看

## 整体流程
- 用 mermaid flowchart 展示核心流程

## 关键实现
- 选 3-5 个关键点深入
- 每点配代码片段（10-30 行，不要整文件贴）

## 性能/边界处理
- 值得注意的优化或边界条件

## 总结
- 可复用的思路
```

**模板 C — 最佳实践/踩坑类：**

```markdown
# [标题]

## 背景

## 实践 1: [标题]
- 问题场景
- 错误做法（配代码）
- 正确做法（配代码）
- 原理

## 实践 2: [标题]
...

## 总结
```

### Step 4: 撰写正文

**写作规范：**

- **语言**：中文为主，技术术语保留英文（如 Facade、Proxy、Singleton）
- **代码块**：始终标注语言（```typescript、```bash），代码 10-30 行为佳
- **文件引用**：使用 `filename.ts:42` 格式标注行号
- **架构图**：优先使用 mermaid，备选 ASCII art
- **段落**：每段 3-5 句，避免长篇大论
- **标题**：层级不超过 3 级（`#` `##` `###`）

**内容要求：**

- 代码示例必须是从实际目录中摘取的真实代码（可以精简，不能编造）
- 每个观点都要有代码佐证
- 不要写"众所周知"、"显而易见"等空洞开头
- 不要在正文中 dump 完整文件内容
- 不要列出所有文件的清单（那不是博客）

### Step 5: 审校输出

写完后逐项检查：

- [ ] 标题包含核心技术关键词（利于 SEO）
- [ ] 第一段能在 3 句话内说明这篇文章讲什么
- [ ] 每个代码块都有语言标注且语法正确
- [ ] 没有超长代码块（>40 行）
- [ ] 至少有一张架构/流程图（mermaid 或 ASCII）
- [ ] 总结部分能独立成段被引用
- [ ] 文件路径和行号准确（已验证过）
- [ ] 没有"TODO"、"TBD"等占位符

## 输出格式

### 输出目录

生成的 Markdown 文件统一保存到本 skill 所在目录（即 `SKILL.md` 同级目录）：

1. 获取当前 skill 的基础目录（系统已提供，无需手动查找）
2. 将生成的 `.md` 文件写入该目录：`<skill-dir>/YYYY-MM-DD-HHmm-<slug>.md`
3. 不要在其他位置散落博客文件

```
<skill-dir>/
  SKILL.md
  2026-05-05-1430-puremvc-cocos-architecture.md
  2026-05-06-0915-module-indexer-guide.md
```

### 文件命名

```
YYYY-MM-DD-HHmm-<slug>.md
```

示例：`2026-05-05-1430-puremvc-cocos-architecture.md`

### 文件 Frontmatter

文件头部添加 frontmatter（适配主流博客平台）：

```yaml
---
title: [文章标题]
date: YYYY-MM-DD
tags: [tag1, tag2, tag3]
categories: [分类]
description: [一句话摘要，用于 SEO]
---
```

## Quick Reference

| 步骤 | 核心动作 | 不要做 |
|------|---------|--------|
| 分析 | 读 3-5 个关键文件，理解架构 | 不要只看文件名就动笔 |
| 选题 | 确定一句话价值 | 不要试图覆盖所有文件 |
| 结构 | 匹配模板（A/B/C） | 不要即兴发挥结构 |
| 撰写 | 中文 + 真实代码片段 | 不要 dump 整个文件 |
| 审校 | 逐项检查清单 | 不要跳过 |
| 输出 | 写入 skill 自身目录 | 不要散落其他位置 |

## Common Mistakes

| 错误 | 为什么 | 正确做法 |
|------|--------|---------|
| 输出目录树作为"架构" | 目录结构不是架构 | 用文字 + 图描述模块关系和职责 |
| 贴整个文件 | 读者不会读 200 行代码 | 摘取关键 10-30 行，解释为什么这段值得看 |
| 面面俱到 | 什么都说 = 什么都没说 | 选一个角度深入，其他的可以另写一篇 |
| 没有流程图 | 纯文字难以表达调用关系 | 至少一张 mermaid sequence/flowchart |
| 英文写作 | 用户明确要中文发布 | 中文为主，术语保留英文 |
| 自说自话 | 没有"问题→方案"主线 | 每节先讲问题再讲方案 |
| 编造代码 | 代码和仓库对不上 | 直接从仓库 copy 后精简 |
| 超过 6 个 ## 章节 | 面面俱到 = 没有重点 | 最多 6 个 ## 节；超出就砍选题范围 |
| 超过 300 行 | 太长的博客没人读完 | 控制在 150-300 行；超出的内容拆成系列文章 |

## Red Flags — 停下来重新审视

出现以下任一信号，说明你正在偏离轨道：

- 文章超过 6 个 `##` 章节
- 代码块超过 40 行
- 目录树直接贴进正文作为"架构"
- 没有 mermaid 或 ASCII 流程图
- 不知道这篇文章的"一句话价值"是什么
- 在 Step 1 没读够 3 个文件就开始写
- "这个文件也很重要，得加一节..."
- 博客文件没有放入 skill 自身目录

**以上任一出现：回到 Step 2，重新确定选题。**（最后一条：停止写作，先把文件移到 skill 所在目录再继续）
