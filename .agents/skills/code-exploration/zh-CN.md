# 代码探索

## 概述

通过从外部触发器追踪执行路径、穿过完整调用栈、映射架构层次并记录依赖关系，深入分析现有代码库功能。目标是理解某物**如何**工作，而不仅仅是**哪些**文件存在。

**核心原则：** 从入口点开始，而不是从文件列表开始。文件列表是准备工作，不是分析。

## 何时使用

- 在实现涉及现有模块的新功能之前
- 当被要求"分析"、"探索"、"追踪"或"理解"代码时
- 在修改某功能之前调查其工作原理
- 当接触代码库中不熟悉的区域时
- 当为将来参考而记录架构时

**不适用于：** 简单的文件查找（直接使用 Grep/Glob）、单函数问题、或已经对代码充分了解的情况。

## 五个阶段

每个阶段完成后再进入下一阶段。

### 阶段 1：入口点发现

找出功能是如何从外部触发的：

- 什么用户操作、系统事件或外部调用启动了这个流程？
- 从触发器追踪到第一层代码
- 识别**所有**入口点，而不仅仅是主要入口
- 标注哪些入口点是生产路径，哪些是遗留/死代码

### 阶段 2：执行路径追踪

追踪从入口到完成的调用链：

- 逐步追踪完整调用栈
- 标注分支逻辑（条件语句、switch 语句、状态机）
- 识别异步边界——每个 `await` 前后发生了什么
- 映射路径上的数据转换
- 标注错误路径和失败模式

### 阶段 3：架构层次映射

识别代码涉及哪些层次以及它们如何通信：

- 是停留在单一层次内还是跨越边界？
- 层次之间如何通信？（事件、通知、直接调用）
- 通信模式是否符合项目约定？
- 标注可复用的边界和反模式

### 阶段 4：模式识别

识别已使用的模式和抽象：

- 出现了哪些设计模式？（单例、有限状态机、观察者等）
- 模式使用是否一致？
- 模块遵循什么命名约定？
- 可见的代码组织原则是什么？

### 阶段 5：依赖文档化

映射模块依赖什么以及什么依赖它：

- 外部库和框架 API
- 内部模块依赖（import）
- 谁消费这个模块？（反向依赖）
- 值得复用的共享工具

## 输出格式

始终使用以下模板生成结构化输出：

```markdown
## 探索：[功能/区域名称]

### 入口点
| 入口点 | 触发方式 | 位置 |
|--------|----------|------|
| [方法/调用] | [如何触发] | [文件:行] |

### 执行流程
1. [步骤 1]
2. [步骤 2]
   - [分支/异步的子步骤详情]

### 架构洞察
- [模式]：[在哪里以及为什么使用]
- [反模式]：[为什么有问题]

### 关键文件
| 文件 | 角色 | 重要程度 |
|------|------|----------|
| [路径] | [功能描述] | 关键/高/中/低 |

### 依赖
- **外部**：[框架 API、库]
- **内部**：[导入的 core 模块]
- **消费者**：[谁导入此模块]

### 新开发建议
- **遵循**：[应采用的模式]
- **复用**：[可用的工具]
- **避免**：[应跳过的反模式]
```

## 常见错误

| 错误 | 修正 |
|------|------|
| 从文件列表开始 | 从入口点开始——问"什么触发了它？" |
| 描述文件而非流程 | 追踪调用链，不要编目文件 |
| 跳过错误路径 | 在映射正常路径的同时映射失败模式 |
| 遗漏反向依赖 | 检查谁导入/消费了这个模块 |
| 缺少建议部分 | 始终包含遵循/复用/避免 |
| 将遗留代码视为活跃代码 | 区分生产路径和死代码 |
| 浅层分析（只读一个文件） | 追踪模块内其他文件的引用 |
| 忽略异步边界 | 标记每个 await 以及之后可能变化的状态 |

## 速查表

| 阶段 | 关键问题 | 产出 |
|------|----------|------|
| **1. 入口点** | 什么触发了它？ | 触发器 + 位置表格 |
| **2. 执行** | 逐步发生了什么？ | 带分支的编号调用链 |
| **3. 架构** | 涉及哪些层次？ | 层次图、通信模式 |
| **4. 模式** | 使用了什么模式？ | 带位置的模式列表 |
| **5. 依赖** | 它需要/提供什么？ | 依赖图 + 消费者 |

## 实际效果

结构化探索可以防止：
- 新代码违反现有模式
- 遗漏已在其他地方处理的边缘情况
- 重复发明已存在的工具
- 破坏层次之间的隐式契约
- 因误解行为而耗费数小时调试
