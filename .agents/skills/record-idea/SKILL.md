# Record Idea

将用户的简要想法记录为 `docs/todo/` 下的结构化文档。只写大框，不展开细节。

## When to Use

- 用户说"记一下这个想法"、"把这个存成 todo"、"记录一下"
- 一个改进点、优化方向、或者还没想清楚但值得留档的概念
- 用户明确说"不用太详细"、"写个大框就行"

**不要用这个 skill 的场景：**
- 正式的功能设计文档 → 走 brainstorming → writing-plans 流程
- 已经想清楚要做的任务 → 直接写 spec/plan

## Process

```
1. 确保 docs/todo/ 目录存在（不存在则创建）
2. 快速理解想法（1-2 句确认即可，不要深挖）
3. 扫描 docs/todo/ 下已有文档，检查是否已存在相同或高度相似的想法
   - 如果已存在 → 告知用户已有相关文档，询问是更新旧文档还是新建
   - 如果不存在 → 继续
4. 生成文件名：docs/todo/<中文名>.md（纯中文，不含日期前缀）
5. 写入文档：第一行为 `YYYY-MM-DD`，第二行起为正文
6. 告知用户文件路径，询问是否需要补充
```

文件名直接从标题取中文名称，中英文混用时用连字符连接。例如"Proxy 自动注册" → `proxy-自动注册.md`，纯中文如"代码混淆" → `代码混淆.md`。

## Document Format

强制存储路径：`docs/todo/<中文名>.md`

```markdown
YYYY-MM-DD

# <标题>

## 动机 / 背景

<1-3 句话说明为什么有这个想法，解决什么问题>

## 大致思路

<核心方案或方向，用表格/列表勾勒框架即可>

## 关键约束 / 注意点

<有什么已知的限制、风险、或者需要提前考虑的点>

## 相关资源（可选）

<相关的文件路径、参考文档、类似实现>
```

每个 section 不超过 5-8 行。用表格呈现对比选项，用列表呈现要点。不写详细实现步骤，不写完整代码。

## Key Principles

- **只写大框** — 动机 + 大致思路 + 注意点，三项齐全即可
- **不写细节** — 不做方案对比、不写具体 API、不画详细流程图
- **快速确认** — 不问一连串澄清问题，1-2 句确认想法核心后直接写
- **可补充** — 文档末尾可以留一句"待补充：xxx"，方便后续回来填
- **去重** — 写入前必须扫描已有文档，避免重复记录同一想法
- **文件名纯中文** — 不加日期前缀，日期放在文档首行

## Anti-Patterns

- 文件名用英文 slug（必须用中文，如 `代码混淆.md` 而非 `code-obfuscation.md`）
- 文件名带日期前缀（日期已写在文档首行）
- 做详细的方案对比（那是 brainstorming 的职责）
- 写完整实现步骤（那是 plan 的职责）
- 追问过多细节（用户就是想先记下来，不是现在做）
- 把想法直接变成代码实现（先记录，后续再决定做不做）
